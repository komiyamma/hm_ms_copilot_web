# プロジェクトの潜在的な問題点

このドキュメントは、`HmMsCopilotWeb` プロジェクトのソースコードから見つかった潜在的な問題点をまとめたものです。

## 1. 自動入力の信頼性が低い

本プロジェクトは、`keybd_event` APIと固定の遅延 (`Task.Delay`) を利用して、キーボード入力をシミュレートしています。

-   **問題点**:
    -   `keybd_event` は古いWindows APIであり、より新しく信頼性の高い `SendInput` APIが推奨されています。特定の条件下で、キー入力が正しく認識されない可能性があります。
    -   `Task.Delay` による固定時間の待機は、実行環境のパフォーマンス（CPU負荷など）によって容易に破綻します。遅延が不十分な場合、入力が速すぎてWebページ側で認識されず、処理が失敗する「レースコンディション」を引き起こす原因となります。

-   **リスク**: ユーザーの環境によって、ツールの動作が不安定になったり、期待通りに動作しなくなったりします。

## 2. クリップボード操作が脆弱である

クリップボードの内容を一時的に保存し、処理後に復元する仕組みになっていますが、この操作にはリスクが伴います。

-   **問題点**:
    -   クリップボードが扱えるデータ形式は多岐にわたります。現在の実装では、一部の形式（画像、ファイルなど）の保存や復元が不完全であり、データが失われる可能性があります。
    -   エラーハンドリングが不十分です。例えば、クリップボードの復元に失敗した場合、ユーザーが元々コピーしていたデータは失われ、このツールが利用したテキストがクリップボードに残り続けてしまいます。

-   **リスク**: ユーザーが意図せず重要なクリップボードデータを失う可能性があります。

## 3. `async void` の不適切な使用

`SendTab`, `SendCtrlV`, `SendReturn` といったメソッドが `async void` として定義されています。

-   **問題点**: `async void` メソッド内で発生した例外は、呼び出し元で捕捉することができず、アプリケーション全体のクラッシュを引き起こす可能性があります。非同期メソッドは、イベントハンドラなどの特殊な場合を除き、`async Task` として定義するのがベストプラクティスです。

-   **リスク**: 予期せぬエラーが発生した際に、ツールが突然終了してしまう可能性があります。

## 4. 全体的なエラーハンドリングの不足

コード全体を通して、エラーハンドリングが最小限に留まっています。多くの `catch` ブロックは空か、一度だけ処理を再試行するだけで、根本的な解決には至りません。

-   **問題点**: エラーが発生した原因を特定するためのログ出力や、ユーザーへのフィードバックがありません。

-   **リスク**: 問題が発生した際に、原因の調査やデバッグが困難になります。

## 5. ハードコーディングされた値

遅延時間（例: `100`ミリ秒）などが、コード内に直接記述されています（マジックナンバー）。

-   **問題点**: これらの値を変更する際に、コードの複数箇所を修正する必要があり、メンテナンス性が低下します。また、なぜその値が設定されているのか、意図が分かりにくくなります。

-   **リスク**: 将来的な機能改修や不具合修正が困難になります。

## 6. 特定の環境への強い依存

このツールは、以下の特定の環境で動作することを前提としています。

-   Windows OS (`user32.dll` のため)
-   Microsoft Copilot のWebページのUI構造

-   **問題点**: MicrosoftがCopilotのUIを少しでも変更した場合（例：入力フィールドの構造変更）、このツールは即座に動作しなくなる可能性が非常に高いです。

-   **リスク**: メンテナンスが継続的に必要となり、少しの外部要因の変化で使えなくなる可能性があります。
